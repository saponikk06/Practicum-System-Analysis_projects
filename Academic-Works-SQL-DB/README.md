# Проектирование и реализация базы данных для учета персональных поездок

**Теги:** `PostgreSQL` `SQL` `Database Design` `Data Normalization` `ERD` `pgAdmin`

Папка содержит учебный проект (курсовую работу) по дисциплине **«Управление данными»**, выполненный в рамках обучения в РГУ им. А.Н. Косыгина. 

## 📌 Описание проекта
Цель работы — разработать реляционную базу данных для структурированного хранения информации о личных поездках, маршрутах и сопутствующих расходах. Система позволяет вести учет пользователей, детализировать этапы путешествий и классифицировать финансовые затраты.

## 🛠 Технологический стек
* **СУБД:** PostgreSQL
* **Среда администрирования:** pgAdmin 4
* **Язык запросов:** SQL (стандарт ANSI SQL)

## 📂 Структура проекта
База данных реализована в схеме `trips` и включает следующие таблицы:
* **`users`** — данные о пользователях системы.
* **`trips`** — заголовки поездок (названия, временные интервалы).
* **`transportation_types`** — справочник видов транспорта.
* **`trip_routes`** — сегменты маршрута внутри одной поездки.
* **`trip_expenses`** — учет расходов, привязанных к конкретным маршрутам.

## 🚀 Ключевые особенности реализации
1. **Целостность данных:** Использование ограничений `FOREIGN KEY` с параметрами `ON DELETE CASCADE` для автоматической очистки связанных данных.
2. **Нормализация:** Структура БД приведена к 3-й нормальной форме (3NF), что исключает избыточность и аномалии обновления.
3. **Бизнес-логика:** Реализованы запросы для агрегации данных (расчет общей стоимости поездки, выборка по типам транспорта и т.д.).

## ⚙️ Системные требования и инструкция по развёртыванию
Для запуска проекта на локальной машине потребуется:
* Установленный **PostgreSQL** (версии 12 и выше).
* Среда для выполнения SQL-запросов (например, **pgAdmin 4** или DBeaver).

**Шаги для развёртывания:**
1. Откройте pgAdmin и создайте новую базу данных (например, `personal_trips_db`).
2. Откройте Query Tool для созданной базы данных.
3. Выполните DDL-скрипт (находится в SQL-файлах проекта) для создания схемы `trips` и таблиц.
4. Выполните DML-скрипты для заполнения таблиц тестовыми данными.
5. База данных готова к выполнению аналитических запросов.

## 📈 Выводы по результатам проекта
В ходе выполнения проекта была успешно спроектирована и реализована масштабируемая реляционная модель данных. Практическое применение нормализации до 3NF позволило избежать дублирования информации о маршрутах и расходах. Настроенные связи и ограничения целостности (CASCADE) гарантируют корректное удаление данных о поездках без появления "осиротевших" записей. Проект демонстрирует уверенные навыки проектирования архитектуры БД и написания DDL/DML запросов.
